#  Cluster CI Validation

This repository includes a GitHub Actions pipeline that validates **all changes** before they are merged into `main`. While automated tools like Renovate are a common source of changes, this pipeline is designed to protect the cluster from **any modification**, whether automated or manual.

## Purpose

The goal of this CI pipeline is to ensure that every change to cluster configuration is **valid, consistent, and deployable**.

It acts as a safety layer for:
- Automated updates (e.g. Renovate)
- Manual changes by contributors (e.g. me)
- Refactors and structural changes

## CI Gate & Automerge Policy

If this CI pipeline fails:
 - No automerge is performed
 - Renovate updates are blocked from merging
 - The pull request is effectively paused
 - A fix is required before progression (typically manual intervention)

This ensures that no change reaches main without explicit validation.

## Workflow Overview

### Main Workflow

**`.github/workflows/cluster_ci.yaml`**

This is the central workflow that orchestrates all validations. It comprises various actions, each with a specific purpose.

### Triggers

The pipeline runs automatically on pull requests and pushes to `main` affecting files in:
- `infrastructure/**`
- `monitoring/**`
- `user-apps/**`
- `scripts/**`
- `.github/workflows/**`

## Validation Jobs

For now, the main workflow delegates validation to three specialized workflows/actions.

### 1. ArgoCD Applications Validation

**Workflow:** `.github/workflows/validate_argocd_apps.yml`

**Script:** `scripts/validate_argocd_apps.sh`

Validates the structure, integrity, and consistency of ArgoCD applications.

#### Checks performed:

- **Duplicate Application Names**
  - Detects conflicts between standalone Applications and those generated by ApplicationSets

- **Sync Wave Continuity**
  - Identifies gaps in sync waves (e.g. missing intermediate values)

- **Kustomization Coverage**
  - Ensures all ArgoCD `Applications` and `ApplicationSets` files are included in `kustomization.yaml`
  - Prevents resources from being silently ignored by ArgoCD

- **Internal Sync-Wave Consistency**
  - Highlights mismatches between Application sync-wave and related resource annotations (e.g. namespaces)

### 2. Kubernetes Manifests Validation

**Workflow:** `.github/workflows/validate_kubernetes_namifests.yml`

Ensures that all Kubernetes manifests can be rendered and are schema-valid.

#### Process:

1. Discover all `kustomization.yaml` files across:
    - `infrastructure/`
    - `monitoring/`
    - `user-apps/`

2. Render manifests using:
   ```bash
   kubectl kustomize --enable-helm path/to/kustomization.yaml
   ```

3. Aggregate all manifests into a single file

4. Validate using `kubeconform`

#### Validation features:

- Strict schema validation
- CRD support via external schema catalogs
- Graceful handling of missing schemas


### 3. Shell Script Validation

**Workflow:** `.github/workflows/validate_shell_scripts.yml`

Performs static analysis on shell scripts.

#### Checks:

```bash
shellcheck -S warning scripts/*.sh
```

#### Behavior:

- Fails if any issues are detected by shellcheck

## Failure Conditions

The pipeline will fail if any of the following occur:

* ArgoCD application conflicts or structural issues are detected
* Kubernetes manifests fail to render or do not pass schema validation
* Shell script validation fails (including warnings reported by `shellcheck`)
